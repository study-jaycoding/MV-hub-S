# 통합 관리 대시보드 — 계획

> 목표: 현재 분리된 **'관리 요약'**(프로젝트 중심)과 **'팀 전체'**(작업자 중심)를 하나로 합쳐,
> 프로젝트 → 에피소드 → 시퀀스 → 컷 계층으로 **소요시간·크레딧·진척도·작업자별 현황**을 한눈에 관리.
>
> 검토: 2026-07-04 (클로드 + 코덱스). 사이드카 원칙(코어 무수정, drop 가능) 유지.

---

## 1. 지금 상태 — 왜 합쳐야 하나 (핵심 발견)

**두 개의 물리적으로 분리된 DB를 읽고 있다.** 이게 통합의 핵심 과제다.

| 화면 | DB | 집계 축 | 한계 |
|---|---|---|---|
| **관리 요약**(ProjectDashboard) | `content_hub.db`(로컬) | 프로젝트별·작업자별 | 내 로컬 데이터 중심 |
| **팀 전체**(TeamOverview) | `manage_hub.db`(서버, `team_generation_fact`) | 작업자별·프로젝트별 | **folder_path(에피/시퀀스) 차원이 없음** |

- 요약 탭은 이미 **에피소드/시퀀스 분석**(`ProjectAnalyticsPanel`: breakdown으로 에피소드별 진척·시퀀스 완료율)을 갖고 있다.
- 팀 전체 탭은 **작업자×프로젝트 매트릭스**는 있지만 에피소드/시퀀스로 못 쪼갠다.
- 즉 "프로젝트별 에피소드/시퀀스당 소요시간·크레딧을 **팀 전체 기준**으로" 는 지금 어느 화면에서도 못 본다.

→ **통합 = 두 DB를 앱 계층에서 `project_id + folder_path`로 합성**(DB를 물리적으로 합치지 않음 — 사이드카 원칙).

## 2. 데이터 지도 — 무엇이 되고 무엇이 부족한가

### ✅ 지금 데이터로 바로 되는 것 (스키마 변경 불필요)
| 원하는 것 | 데이터 출처 |
|---|---|
| 프로젝트별 생성수·완료·게시·크레딧·소요시간 | `dashboard_summary`(content) / `team_overview`(팀) |
| 작업자별 생성·크레딧·소요시간 | 양쪽 `by_worker` |
| 작업자 × 프로젝트 매트릭스 | `matrix` |
| 에피소드/시퀀스 계층 | `generation.folder_path` split("/") → segs[0]=에피, segs[1]=시퀀스 (검증됨) |
| 프로젝트 계획 대비 실적 | `project_planning`(budget_credits·due_date) vs 실제 |
| 시퀀스별 완료율 | `breakdown`의 final/count |

### 🔷 계산으로 파생 가능 (엔드포인트만 추가)
| 파생값 | 방법 |
|---|---|

| 크레딧 커버리지(real/est/unknown 비율) | credit_source 집계 |

| 진행 상태 분포 | generation.status / is_final / is_shared |

### 🔴 부족 — 보강 필요 (탐색으로 확인된 구멍)
| # | 부족한 것 | 영향 | 보강안 |

## 3. 통합 설계

### 엔드포인트 — 하나로 합성
```
GET /api/manage/dashboard?project_id=&from=&to=&worker=
```
응답(코덱스 설계):
```
{
  totals,          // 총 크레딧·소요시간·완료율·예산대비
  projects[],      // 프로젝트별 요약
  hierarchy[],     // project > episode > sequence > cut  (folder_path 롤업)
  worker_matrix[], // 작업자 × (프로젝트|시퀀스)
  timeseries[],    // 기간별 크레딧·작업량
  alerts[],        // 예산초과·마감초과·담당없음·커버리지낮음
  coverage         // real/est/unknown, elapsed 커버리지, confidence
}
```
- 팀 실적(team_generation_fact) + 계획/작업(project_planning·project_task)을 **앱 계층에서 project_id+folder_path로 합성**.
- `coverage`/`confidence`를 **반드시 포함** — elapsed는 허브 생성분만, 크레딧은 60초 매칭이라 "측정 N/전체 M"을 항상 노출(기존 요약 탭 원칙 계승).

### 화면 레이아웃
```
┌─────────────────────────────────────────────────────────┐
│ [필터] 프로젝트 · 기간 · 작업자        [KPI] 크레딧 · 시간 · 완료율 · 예산대비 │
├──────────────────────────────┬──────────────────────────┤
│ 중앙: 계층 트리 테이블          │ 우측: 작업자별 진척도       │
│  프로젝트 ▸ 에피소드 ▸ 시퀀스    │  담당·생성수·크레딧·시간    │
│   각 층: 소요시간·크레딧·진척%·상태│  최근 활동                 │
├──────────────────────────────┴──────────────────────────┤
│ 하단: 기간별 크레딧·작업량 추이 막대                        │
├─────────────────────────────────────────────────────────┤
│ 경고: ⚠ 예산초과 · 마감초과 · 담당자없음 · 커버리지 낮음      │
```

- **중앙 계층 트리**가 핵심: 프로젝트를 펼치면 에피소드, 에피소드를 펼치면 시퀀스, 각 층에 소요시간·크레딧·진척도·상태 롤업.
- 행 클릭 → 그 스코프로 우측·하단 필터 연동(기존 TeamOverview 필터 UX 계승).

## 4. 단계별 계획 (코덱스 권장 순서)

### Phase 1 — MVP (스키마 변경 없음, read-only 합성)
1. `GET /api/manage/dashboard` 신설 — 기존 dashboard_summary·team_overview·breakdown·matrix를 **한 응답으로 합성**.
2. breakdown에 elapsed 합 추가, team 집계에 folder_path·status 축 추가(#1·#2·#6 — 컬럼 이미 있음).
3. 프론트: 요약+팀전체를 **한 탭으로 합친 대시보드** 컴포넌트. 중앙 계층 트리 + 우측 작업자 + 하단 추이 + 경고.
4. `coverage`/`confidence` 노출.
→ **이 단계로 Jay가 원하는 것의 80%가 나온다**(진척도는 완료율 기준).

### Phase 2 — 진척도·계획 보강 (스키마 추가)
계층/진척 의미가 확정된 뒤:
- **선택 A(가벼움)**: `project_task`에 컬럼 추가 — `planned_count`·`weight`·`budget_credits`·`estimated_seconds`. 진척도 = 완료/planned.
- **선택 B(정밀)**: `project_work_unit` 신설 — project/episode/sequence/cut 을 명시 엔티티로(status·assignee·budget·due·planned_count·weight). folder_path 파싱 의존을 줄이고 층별 예산·일정·담당 가능.

### Phase 3 — 성능 (규모 커지면만)
- 프로젝트당 생성물 수만 건 넘으면 `team_generation_fact(project_id, folder_path)` 인덱스 + 롤업 캐시.

## 6. 상세 설계 (코덱스 정밀화)

### 6-1. `project_work_unit` — 관리단위 정본 (사이드카 신설)
기존 `project_task`(폴더 자동 작업)와 겹치므로, **work_unit을 장기 정본으로 두고 기존 task는 호환 계층으로 단계 이행**(완전 대체 아님·영구 공존 아님 — 이중 원천화 방지).

```
project_work_unit(
  id, project_id,
  kind,               -- episode | sequence | cut | task
  parent_id,          -- 계층(episode→sequence→cut)
  key, name,
  folder_path,        -- 자동 파생 unit의 폴더 경로(수동 unit은 NULL)
  source,             -- auto_folder | manual
  status,             -- 진척 정본(수동 수정은 leaf/sequence 중심)
  assignee_uid,       -- 배정(누가 맡았나) — creator_uid(누가 생성)와 별개
  start_date, due_date,
  planned_count,      -- 계획 컷 수(진척·계획대비 분모)
  budget_credits, estimated_seconds,  -- 층별 예산·예상시간
  progress_weight,    -- 롤업 가중치
  sort_order, note, description,
  created_at, updated_at, last_seen_at
)
-- 예정 생성자(복수) — 이미 구현된 task_planned_creator 를 이행. assignee_uid(단수 리드)와 별개 축.
work_unit_planned_creator(
  unit_id, creator_uid, added_by, created_at,
  PRIMARY KEY(unit_id, creator_uid)
)
```
- `assignee_uid` = PM 배정/리드(단수). `work_unit_planned_creator` = self-assign 실무 예정자(복수).
  실제 생성자(creator_uid)는 work_unit 컬럼이 아니라 **연결 생성물에서 파생**(자동).
- 이행: `project_task.assignee_uid`→work_unit, `task_planned_creator.task_id`→`work_unit_planned_creator.unit_id`,
  `task_generation`→`work_unit_generation`. 기존 `/tasks` API/UI는 당분간 work_unit을 task 모양으로 반환(회귀 방지).
- **사이드카 원칙 유지**: 코어 무수정, drop 한 번에 제거 가능.

### 6-2. folder_path ↔ work_unit 동기화 (자동+수동 혼합)
- `folder_path`에서 episode(segs[0])/sequence(segs[1]) 파싱 → **create-only 자동 unit 생성**(새 폴더 생기면 추가).
- **PM 수정값(name/status/assignee/budget/due)은 절대 자동 덮어쓰기 금지.**
- 폴더가 사라져도 unit 삭제 금지 → `last_seen_at`/`gen_count=0`으로 **stale 표시**.
- 수동 unit은 `folder_path=NULL, source=manual`, 자동 sync 대상 제외.
- 파싱 실패 → `(미지정)` 버킷 + PM이 수동으로 folder_path 연결 가능. (복잡한 규칙 엔진은 과설계 — 필요 시 나중에 `folder_parse_rule` sidecar.)

### 6-3. 진척도 롤업 (작업상태 기준)
- **정본 = `work_unit.status`** (`generation.status`는 제작 실행 상태라 관리 진척에 안 씀 — 자동 제안엔 참고 가능).
- 표시 우선순위: **① 상태 분포 스택바**(완료3·게시2·진행4·시작전1) → ② 계산 progress % → ③ 대표 rollup status.
- progress 가중치(초안, 업무 합의 필요): `not_started 0 / pending 0.15 / in_progress 0.5 / publish 0.8 / done 1 / omit=분모제외`.
- 상위(에피소드/프로젝트) progress = `SUM(child_progress × child_weight) / SUM(child_weight)`.
- **parent는 rollup_status만 표시**(수동 수정은 leaf/sequence 중심 — parent 수동값과 롤업 충돌 방지).

### 6-4. 지표 정의 (층별, 중복집계 방지)
| 지표 | 정의 |
|---|---|
| 소요시간 | 하위 생성물 `elapsed_seconds` 합 (커버리지 `known N/M` 병기) |
| 크레딧 | 하위 `real_credits`(없으면 `est_credits`) 합 (`matched N/M` 병기) |
| 진척도 | 하위 `work_unit.status` 롤업(6-3) |
| 예산대비 | `work_unit.budget_credits` vs 실제 크레딧 |
| 계획시간대비 | `work_unit.estimated_seconds` vs 실제 elapsed |
| 계획수대비 | `planned_count` vs 실제 생성물/완료 unit 수 |

- **★생성물은 가장 깊은 unit에 1회만 귀속**, 상위는 rollup만 → 중복 집계 없음.
- planned_count·budget·estimated는 모든 층에 둘 수 있되, 없으면 children 합으로 파생.

**신원 3축 지표(예정 생성자 활용)** — 진척률 자체는 status 롤업(불변), 예정은 "누가·개인 작업량"의 분모:
| 지표 | 정의 |
|---|---|
| 개인 예정 작업량 | planned_creator로 들어간 unit 수 / planned_count |
| 개인 진척 | planned_creator가 포함된 unit들의 status 롤업 |
| 예정대비실제 | planned_creator 집합 vs actual creator 집합(불일치=관리 신호) |
| 배정대비예정 | assignee_uid vs planned_creator 집합 |
| 무예정 작업 | planned_creator 비어 있는 작업(담당 미정) |
| 미착수 예정 | planned_creator는 있는데 status=not_started |

- 초기 규칙: 복수 planned creator면 각 사람에게 1건씩 카운트(가중치·지분율은 나중).

### 6-5. 레이아웃 (확정)
**중앙 계층 트리 테이블 컬럼**:
```
이름 | 상태·진척 | 담당 | 마감 | 계획수/실적 | 크레딧 실제/예산 | 소요시간 실제/예상 | 최종·게시 | 커버리지 | 최근활동
```
층별 노출:
- **프로젝트**: 전체 progress·예산/실적·마감·상태(active/done/hold)·경고
- **에피소드**: 시퀀스 수·rollup progress·담당 lead(or mixed)·예산/실적
- **시퀀스**: status·assignee·planned_count·gen/final/shared·크레딧·elapsed·마감
- **컷**: expand 시 상세 패널(썸네일·댓글·연결 생성물)

**우측 "참여자" 패널 — 하나로 통합, 선택 계층에 따라 전환** (신원 3축):
- **프로젝트 행 클릭** → 배정 멤버 전원 + 역할(PM/Supervisor/Creator) + 프로젝트 전체 배정/예정/생성 집계 수. 원천 `project_member`.
- **시퀀스 행 클릭** → 그 시퀀스의 `assignee`+`planned_creators`+실제 `creator_uid`를 합친 참여자 리스트.
- **에피소드 행 클릭** → 하위 시퀀스의 참여자 요약.
- **배지 3종**(한 사람이 여러 축 가능): `배정`(assignee) / `예정`(planned_creator) / `생성`(actual). 예: "Jay 예정·생성", "Risa 배정", "Mina 생성".
- ★불일치가 관리 신호: "A 예정인데 B가 생성", "배정만 있고 예정 없음" 등을 배지 조합으로 드러냄.
- **작업탭 연동(구현 완료)**: 담당 select(48d5373) + 예정 생성자 self-assign·엑셀식 셀 복사(1b3ad6e·7df1641). 대시보드는 이 데이터를 재사용.

## 8. 남은 리스크 / 열린 결정
- **task↔work_unit 이행기**: 이중 원천화되면 상태 불일치 → 이행 기간 짧게, 호환 API로 단일 원천 유지.
- **folder_path 파싱은 추정** → PM 수동 연결·수정 경로 필수.
- **progress 가중치는 업무 합의 필요** → 초기엔 상태 분포 스택바를 더 강하게(% 는 보조).
- **두 DB 동기화 타이밍**(team_generation_fact vs 로컬 task/planning) → "마지막 동기화 시각" 표시 필요.
- **엑셀식 붙여넣기 권한**: `replace`=PM(manage), `add+본인만`=일반 멤버(read). 프론트가 복사 내용이 '나뿐'이면 add 로 자동 판정(구현됨) — 일반 멤버 대량 self-assign 이 403 안 나게.
- **3축 정합**: 대시보드가 배정/예정/생성 3축을 반영 안 하면 작업탭과 관리 화면 의미가 다시 어긋남 → Phase 1 응답에 셋 다 포함.
- **added_by remap**: `task_planned_creator.added_by` 는 현재 audit 스냅샷. 나중에 집계·권한에 쓰면 remap 대상 추가 필요(지금은 불필요).
- **열린 결정**: work_unit을 새 테이블 정본으로 갈지(권장) vs project_task 확장으로 갈지 최종 확정.
