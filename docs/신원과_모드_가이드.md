# 신원(Identity)과 모드(Mode) 가이드 — 새 코드 작성 전 필독

> 이 문서는 "어떤 상황에 어떤 신원 헬퍼를 쓰는가"를 고정하는 **단일 진실**이다.
> 신원을 직접 다루는 코드(권한·소유권·스코프·표시·실시간)를 짜기 전에 이 표를 먼저 본다.
> `creator_uid`·`worker_id`·`email` 을 **직접 접근해 새 로직을 만들지 말 것** — 아래 헬퍼를 쓴다.
>
> 최종 갱신: 2026-07-04. 근거: `backend/app/deps.py`, `backend/app/repo/identity.py`, `docs/ARCHITECTURE.md`.

---

## 1. 신원 축(axis) — 5가지, 절대 섞지 말 것

| 축 | 형식 예 | 의미 | 저장 위치(예) |
|---|---|---|---|
| **email** | `oz1@millionvolt.com` | 로그인 계정(PK). 사람 신원의 앵커 | `account.email` |
| **creator_uid (실제)** | `user_36s5...` | 힉스필드 실제 생성자. 작성자·소유권 축 | `generation.creator_uid` |
| **creator_uid (합성)** | `acct:oz1@...` | 실제 uid 미확정 시 이메일 앵커 **임시** 신원. push 후 실제 uid로 remap | 위와 같은 컬럼 |
| **worker_id** | `me` | **워크스테이션(작업 PC)** 축. ★작성자가 아님 | `generation.worker_id` |
| **name** | `오즈` | 표시이름(UI 전용). uid·email 은 UI에 절대 노출 금지 | `creator.name`, `account.name` |

**왜 축이 이렇게 많은가**: 허브 로그인 계정(브라우저)과 그 PC의 힉스필드 CLI 인증(생성 주체)이 **완전히 별개**이기 때문이다. "허브엔 oz1로 로그인했는데 그 PC의 CLI는 jay" 가 정상 상황 → account(로그인)와 creator(작성자)를 분리하고 `account.creator_uid` 로 연결한다. (근거: `docs/AI_CONTEXT.md §4`, `docs/ARCHITECTURE.md §8`)

## 2. 헬퍼 선택표 — 무엇을 언제 (모두 `backend/app/deps.py`)

| 하려는 것 | 쓸 헬퍼 | 반환(AUTH on) | 반환(AUTH off) |
|---|---|---|---|
| **권한/소유권 판단** (이 리소스가 내 것인가, 이 행동을 할 수 있나) | `account_actor_uid(request)` | `creator_uid` 또는 `acct:email` (없으면 409) | `None` |
| **쓰기 주체 신원** (author·shared_by·created_by 컬럼에 넣을 값) | `actor_id(request)` | 위와 같음 | `'me'`(DEFAULT_WORKER_ID) |
| **쿼리 스코프** ('내 작업'·'내 facet'·'내 job'만 걸러낼 때) | `account_scope_uid(request)` | `creator_uid`, 미링크면 `acct:email`, 그도 없으면 `'\x00'`(불가능값) | `None`(전체 가시) |
| **실시간(WS) 채널키** (progress·synced 를 누가 받을지) | `realtime_scope(acc)` | `acct:email`(email 기반) | `None`(단독=전체) |
| **표시이름** (카드·사이드바·코멘트·멤버 이름) | `identity.resolve_display_names(conn, uids)` | `creator.name → account.name → email 로컬파트 → None` | 동일 |

### 왜 스코프 헬퍼가 서로 다른가 (헷갈리기 쉬운 지점)
- `account_scope_uid` 는 **쿼리 필터 전용** — 미링크 계정이 `None` 으로 떨어져 필터가 풀리면 남의 데이터가 다 노출되므로 `'\x00'`(아무 행도 안 맞는 값)으로 막는다. **actor/권한 판단엔 쓰지 말 것**(`'\x00'` 은 본인 것에도 안 맞아 깨진다).
- `realtime_scope` 는 **WS 채널키 전용** — `creator_uid` 가 아니라 **email 기반**이다. 이유: `acct:→user_` remap 중에도 스코프가 안 바뀌어 열린 소켓이 계속 자기 알림을 받고, `creator_uid` NULL 계정끼리 같은 버킷을 공유해 progress 가 새는 것도 막는다. **DB에 저장되지 않는다**(채널 매칭에만 쓰는 임시 키).
- 즉 같은 계정이 권한에선 `user_X`, WS에선 `acct:email` 로 식별될 수 있다 — **의도된 비대칭**이고, WS 스코프는 저장 안 되므로 섞이지 않는다.

## 3. 불변식 — 하지 말 것 (위반 = 실측 버그)

1. **`worker_id` 로 작성자를 판별하지 말 것.** `generation.worker_id` 는 항상 `'me'`(워크스테이션)다. 작성자는 `creator_uid`(또는 공유는 `share.shared_by`)로 판별한다. (근거: `generations.py` 목록 쿼리 주석)
2. **표시이름은 `resolve_display_names` 로만.** 다른 곳에서 `creator.name`·`account.name` 을 직접 조회해 이름을 만들지 말 것(해석 규칙이 갈라진다). 이름 변경은 두 소스를 함께 갱신해 읽기 시점에 전파된다.
3. **계정↔creator 재연결 오염 금지(3중 불변식).** ① 잡은 자기 고유 `creator_uid` 유지(uid 없을 때만 보강) ② 계정이 이미 실제 uid에 연결됐으면 **재연결 금지** ③ 에이전트는 "로컬 **전체** 목록의 최다 `user_<id>`"를 명시 전송(fresh 부분집합만 보면 남의 레퍼런스에 오염). (근거: `ARCHITECTURE.md §7`, `AI_CONTEXT.md`)
4. **`acct:email` → `user_` 전환은 `remap_creator_uid` 로 전 테이블 정합.** 신원 컬럼을 새로 추가하면 `identity.py:_REMAP_PLAN` 에 등재할 것. 메인 DB 밖(별도 trash DB·JSON payload)의 신원은 자체 경로에서 재해석한다(현재 `trash.py` B+ 방식).
5. **`job_id` 를 식별 앵커로 쓰지 말 것.** id=uuid 가 안정 앵커, `job_id` 는 동기화 멱등용 속성(UNIQUE). (근거: `DESIGN_id_unification.md`)

## 4. 모드(Mode) — 조합이 복잡도의 근원

| 모드 | `CONTENT_HUB_AUTH` | 설명 |
|---|---|---|
| **로컬 허브(단독)** | off(0) | 내 PC·내 것. 계정 격리 없음. 신원 헬퍼는 대부분 `None`/`'me'` 반환 → 전체 가시. |
| **공유 서버(다계정)** | on(1) | 여러 계정. 신원 헬퍼가 계정별로 격리. RBAC 게이트가 실제로 강제됨. |
| **프록시(위임)** | off + shared_server_token | `proxying()` — 로컬 허브가 데이터 요청을 공유 서버로 중계. `/api/manage/*` 등이 서버로 감. `/api/auth/config` 는 로컬 전용(프록시 안 함). |

- **모든 RBAC·권한 가드는 AUTH off 면 통과**(식별 먼저, 차단은 서버 모드에서). 새 권한 코드는 이 규칙을 따른다.
- **서버는 힉스필드 CLI 를 돌리지 않는다**(push 모델). 생성은 각자 로컬 CLI, 서버엔 메타만 push. 서버는 "이 job이 진짜 누구 것인가"를 검증할 수단이 없어 **push 데이터를 신뢰**한다(내부 팀 신뢰 모델).
