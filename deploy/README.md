# 배포 — 앞단 리버스 프록시(선택, Phase 2 미디어 분리 서빙)

평소엔 `MV_server.bat` 하나로 충분하다(백엔드가 프론트·미디어·API 를 한 오리진에서 서빙).
**미디어가 수만 건으로 늘고 동시 다운로드가 많아질 때**만 앞단에 nginx/Caddy 를 두어
정적 파일(/media·번들)을 분리 서빙하면 uvicorn 이 정적 전송에 안 막힌다.

## 핵심 — 코드 무변경
프론트는 이미 상대경로(`/api`·`/ws`·`/media`)라, 프록시가 같은 오리진을 유지하면
**프론트엔드·백엔드 코드 변경이 전혀 없다.** 프록시는 `/media`·정적은 디스크에서
직접, `/api`·`/ws` 만 백엔드(127.0.0.1:8010)로 넘긴다.

## nginx
1. [nginx.conf](nginx.conf) 의 경로(`D:/ClaudeCode/...`)를 실제 위치로 확인.
2. `MV_server.bat` 로 백엔드는 8010 에 그대로 띄운다.
3. `nginx -c <nginx.conf 절대경로>` 실행.
4. 접속: `http://<이 PC>:8080` (8010 은 nginx 뒤로 숨김).

## Caddy (더 간단, 자동 HTTPS)
[Caddyfile](Caddyfile) 참고. `caddy run --config Caddyfile`.

## ⚠️ 인증(CONTENT_HUB_AUTH=1)과 함께 쓸 때
백엔드는 `/media` 도 세션으로 보호하지만, 프록시가 `/media` 를 직접 서빙하면 그 보호가
우회된다. AUTH 를 켤 거면 프록시의 `/media` 블록을 빼서 **백엔드가 /media 를 서빙**하게
두거나(분리 효과는 포기), `auth_request` 로 백엔드 세션 검증을 붙여라.
기본값(AUTH off)에서는 그대로 써도 된다.

## 트레이드오프
- 장점: 정적 전송이 uvicorn 워커를 안 먹음 → 동시 다운로드·썸네일 폭주에도 API 응답 일정.
- 비용: 프록시 1개를 추가로 띄워 운영해야 함(프로세스·설정 관리). 데이터가 적을 땐 불필요.
